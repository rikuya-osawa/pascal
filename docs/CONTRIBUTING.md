# PASCAL への貢献ガイド

「日本語圏初の体系的なメンタルモデル辞書」を共に創り上げてくださる皆さんの力を必要としています。
本プロジェクトは、単なる用語集ではなく、複雑な世界でより良い判断を下すための「知的OS」を目指しています。

## 🤝 貢献の種類

1. **モデルの新規追加**: 未掲載のメンタルモデルの執筆。
2. **コンテンツのブラッシュアップ**: 既存モデルの誤用例、実践例、関連モデルの追記。
3. **多言語展開の推進**: 自然な英語表現への修正、または日本語記事の英訳。
4. **技術的貢献**: UI/UXの改善、検索ロジックの最適化、パフォーマンス向上。

## 📝 コンテンツ執筆定義表（Front Matter）

モデルの品質と一貫性を保つため、以下の定義に基づいて Front Matter を記述してください。

### 1. Nature（性質）の選択基準

ユーザーがモデルを「どう使うべきか」を直感的に理解するための分類です。

| Nature        | 定義           | 判定基準                                 | 代表例                                                                     |
| ------------- | -------------- | ---------------------------------------- | -------------------------------------------------------------------------- |
| **Framework** | **型・構造**   | 情報を整理するための「静的」な器。       | 紙とペンで図を描き、空の箱を埋める作業が発生するか？                       |
| **Cognition** | **認知・視点** | 世界を解釈するための「能動的」なレンズ。 | 脳内で「意識的に視点を切り替えよう」とプロセスを回すものか？               |
| **Principle** | **法則・真理** | 抗えない「客観的」な世界の性質・真理。   | 人間の意思に関わらず「世の中はそもそもこうなっている」と認めるべき現象か？ |

> **迷った時の「分かち書き」指針**
>
> - **概念そのものか、手法か**: 例えば「反脆弱性」は戦略（Cognition）ではなく、世界の性質（Principle）として定義します。
> - **主たる価値は何か**: 整理ツールとしての側面が強ければ **Framework**、思考のOSをアップデートするものであれば **Cognition** を優先してください。

### 2.Methodology（アプローチ）の定義

PASCALでは、各メンタルモデルが「思考のどのフェーズで機能するか」を以下の5つの **Methodology** で定義しています。モデルを新規追加・修正する際は、この定義に基づいたタグ付けを行ってください。

| フェーズ   | Methodology     | 役割（エンジンの機能）                 |
| :--------- | :-------------- | :------------------------------------- |
| **広げる** | **Generative**  | 無から有を作る、枠を外す。             |
| **整える** | **Structural**  | 複雑さを分解し、関係を可視化する。     |
| **疑う**   | **Critical**    | 本質を突き、リスクやバイアスを暴く。   |
| **決める** | **Decisive**    | 優先順位を確定し、行動へ移す。         |
| **繋げる** | **Interactive** | 他者の脳を借り、チームとして実行する。 |

### 📝 運用ルール

- 各モデルに対し、該当する手法を **1〜3つ** 割り当ててください。
- 迷った場合は、そのモデルが「PDCAサイクルのどの段階で最も威力を発揮するか」を基準に判断します。
  - **PLAN (設計)**: Generative / Structural / Critical
  - **DO / CHECK (実行・検証)**: Interactive / Decisive
  - **ACT (改善)**: Critical / Structural

### 3. タグ（Tags）の定義

PASCALでは、モデルを「学術領域」や「特定の理論体系」で横断検索するために以下のタグを使用します。
各モデルには、該当するタグを制限なく付与できます。

#### Core Domains（基本領域）

実務や学問の主要なカテゴリです。

| タグ              | 名称 (JP / EN)                     | 対象とするモデルのイメージ                   |
| :---------------- | :--------------------------------- | :------------------------------------------- |
| **Strategy**      | 戦略 / Strategy                    | 競争優位、環境分析、中長期的な方向性。       |
| **Marketing**     | マーケティング / Marketing         | 顧客理解、価値提案、市場ポジショニング。     |
| **Management**    | 組織・PJ管理 / Management          | チーム運営、進捗管理、リソース最適化。       |
| **Psychology**    | 心理学・行動経済学 / Psychology    | 人の思考の癖、認知バイアス、意思決定の心理。 |
| **Statistics**    | 統計 / Statistics                  | 確率、データ分析、数学的アプローチ。         |
| **Economics**     | 経済学 / Economics                 | 合理性、資源配分、市場の原理、トレードオフ。 |
| **Productivity**  | 生産性 / Productivity              | 個人の習慣、時間管理、タスクの最適化。       |
| **Innovation**    | 新規事業 / Innovation              | ゼロイチ、仮説検証、不確実性への挑戦。       |
| **Communication** | コミュニケーション / Communication | 伝達、合意形成、論理的な説明、対人関係。     |

#### Meta-Tags（知の巨大体系）

多くの個別モデルを内包する、特に大きなトピックスです。

| タグ                 | 名称         | 定義                                             |
| :------------------- | :----------- | :----------------------------------------------- |
| **Systems-Thinking** | システム思考 | 要素間の「つながり」や「ループ」を重視する体系。 |
| **Lean**             | リーン       | 「無駄の排除」と「学習の高速化」を軸とする思想。 |
| **Game-Theory**      | ゲーム理論   | 「他者の意思決定との相互作用」を分析する理論。   |

---

#### 📝 タグ選定のガイドライン

- **Core Domain** から少なくとも1つ、必要に応じて **Meta-Tags** を選択してください。
- **Methodology**（Structural, Critical等）が「どう考えるか」を指すのに対し、**Tags** は「何についての知見か」を指します。
- 新しい Meta-Tag（例：デザイン思考、行動経済学の深掘り等）が必要だと感じた場合は、まず `constants/tags.ts` への追加を検討してください。

### 4. Score（評価スコア）

各項目 **1（低）〜 5（高）** で評価します。

| 項目           | 1 の目安                       | 5 の目安                               |
| -------------- | ------------------------------ | -------------------------------------- |
| **応用度**     | 特定の専門職や場面のみで有効   | あらゆる日常・仕事の場面で有効         |
| **即効性**     | 成果が出るまで数年〜一生かかる | 知った直後の判断からすぐに使える       |
| **理解難易度** | 直感的にすぐ理解できる         | 数学・哲学的な深い学習を要する         |
| **誤用リスク** | 悪影響がほとんどない           | 誤用が致命的な判断ミスを招く恐れがある |

### 5. Sources（出典・参照）

情報の信頼性と透明性を担保するため、参照資料を `sources` フィールドに集約します。
当PASCALは既存の概念をベースにしつつも、**実務経験に基づき独自の解釈で再構成したもの**です。出典は「転載元」ではなく、あくまで「知見の裏付け（Reference）」として扱います。

#### 記述形式

```yaml
sources:
  - title: "文献またはサイトのタイトル"
    author: "著者・団体名"
    url: "https://..." # 任意
    sourceType: "primary" # primary(原典) | reference(参照) | inspiration(着想元)
    sourceLanguage: "ja" # ja | en
    description: "補足（例：定義の着想元として参照）" # 任意
```

#### `sourceType` の定義

- **primary**: **原典・提唱者**。その概念が初めて定義された書籍や論文。
- **reference**: **参照資料**。信頼できる第三者による解説、公式ドキュメント、辞書的定義。
- **inspiration**: **着想元**。独自の再構成を行うにあたり、ヒントや事例を得たリソース。

## 🛠 執筆の共通ルール

- **アンチパターンの記載**: 読者が「わかったつもり」で失敗しないよう、実際の現場で起きがちな誤用例や誤解を最低1つは記載してください。
- **日英並行展開**: 本プロジェクトはグローバルな知の共有を目指しています。日本語版を作成した際は、可能な範囲で英語版（`en/` フォルダ）への移植も検討してください。

## 🚀 貢献のワークフロー

1. **Issue の確認**: 重複を避けるため、既存の Issue や PR を確認してください。
2. **Branch 作成**: `feature/add-[model-name]` 形式のブランチで作業を開始してください。
3. **Pull Request**: テンプレートに従って PR を作成してください。
